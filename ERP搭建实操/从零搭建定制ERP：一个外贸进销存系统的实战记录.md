# 从零搭建定制ERP：一个外贸进销存系统的实战记录

这段时间，我从零开始做了一套适合自己业务的外贸进销存 ERP，取了一个内部项目名。

最开始的需求其实很朴素：我需要一个能登记供应商、管理采购和销售、统计收付款、管理库存、保存客户单据的系统。市面上的 ERP 很多，但要么太重，要么字段不贴合自己的外贸业务流程。于是决定做一套本地 ERP，先满足真实业务，再逐步优化。

## 一、最初的目标

一开始定下的核心功能包括：

- 供应商档案管理
- 商品库存管理
- 采购入库、销售出库
- 应收、已收、应付、已付统计
- 客户档案和客户单据
- 多文件、多图片上传
- 采购/销售图片识别录入
- 本地运行，可通过 Tailscale 给多台电脑访问
- 后期可以由 Codex 持续维护和升级

这个阶段的重点不是“做得多漂亮”，而是先把业务闭环跑通。

## 二、从桌面 ERP 开始

最早做的是本地桌面启动方式。ERP 运行在 Windows 本机，通过 PowerShell 启动一个本地服务，然后浏览器访问：

```text
本地访问地址
```

后来为了让办公室电脑、家里电脑都能访问同一套数据，又加入了 Tailscale 内网访问方式：

```text
内网访问地址
```

这样系统不用暴露到公网，也能在不同电脑上登录同一套 ERP。

## 三、逐步建立业务模块

系统最初只有几个基础模块，后来慢慢扩展成完整的业务结构。

目前主要模块包括：

- 经营总览
- 供应商管理
- 商品库存
- 采购与销售
- 出入库流水
- 客户管理
- 客户单据
- 收付款管理
- 进项发票统计
- 台账
- 邮件
- 审计日志
- 系统自检

我后来发现，ERP 的关键不是页面数量，而是数据关系是否顺畅。

比如一张销售单，不应该只是记录“卖了什么”，还应该能继续追溯：

```text
销售单 → 客户 → PI → 产品 → 供应商 → 成本 → 收款 → 库存 → 台账利润
```

这才是 ERP 真正有价值的地方。

## 四、供应商和报价表

供应商模块一开始只是登记名称、联系人、电话和地址。

后来加了供应商报价表，每个供应商下面可以保存多份报价表，报价表可以上传原件图片，也可以把识别出来的产品、型号、单价保存成表格。

这个功能对采购很有用，因为后面录采购单时，可以直接引用历史商品和供应商报价。

后来又做了这些优化：

- 报价表可以自定义名称
- 已有报价表可以修改名称
- 报价原件支持多个文件
- 产品搜索容错率提高
- 货号字段改成“型号”
- 供应商界面直接显示应付、已付、未付金额

## 五、客户管理和客户单据

客户管理是后来改动最多的地方。

一开始客户档案、客户合同、单据库是分散的，实际用起来很乱。后来把它们合并成“客户管理”下的一套关系：

```text
客户档案 → 客户合同 → 客户单据 → 订单/CI No.
```

客户单据又按业务分成：

- 报关订单
- 非报关订单

报关订单下面继续分：

- 客户清关单据
- 国内清关单据
- 订单附件

客户清关单据包括：

```text
PI、PL、CI
```

国内清关单据包括：

```text
CI、PL、contract、国内采购合同、国内采购发票、国内运费发票、国际运费发票、报关单、报关委托书、放行书、出口退税联、报关费发票、提单、收汇水单、结汇水单、出口发票
```

最近又新增了“订单附件”分组，用来上传打包照片、发货照片、售后照片，方便以后售后备查。

这个功能很实用，因为很多外贸售后问题最后都要回头查包装、装箱、发货现场照片。

## 六、采购、销售和库存闭环

采购和销售是 ERP 的主干。

采购入库需要填写：

- 供应商
- PI No.
- 产品名称
- 型号
- 颜色
- 数量
- 单价
- 总金额
- HS Code
- 税点
- 不含税金额
- 含税金额

销售出库需要填写：

- 客户名称
- 客户信息
- 币种，人民币或美元
- 产品名称
- 型号
- 颜色
- 数量
- 单价
- 总金额
- HS Code
- 产品对应供应商

销售单还可以自动生成 PI，并且 PI 可以预览、修改、下载。

库存方面，系统会自动同步：

```text
采购入库 → 库存增加
销售出库 → 库存减少
手动调整 → 生成库存流水
```

后来进一步优化了业务录入闭环：

- 采购选择供应商后，商品下拉只显示该供应商的商品
- 选择商品后自动带出型号、颜色、HS Code、供应商、价格
- 销售单每个产品行都保存供应商来源
- 库存流水也记录客户、供应商、业务日期

这样后续查账时就不会断链。

## 七、收付款和台账

收付款模块后来也做了很多调整。

现在支持：

- 应收
- 已收
- 应付
- 已付
- 人民币
- 美元
- 客户/供应商
- 日期
- 多文件上传
- 关联订单
- 按客户、供应商、销售单分组
- 年/月汇总

台账模块主要按 PI 汇总订单利润，字段包括：

- PI 号
- 客户名称
- 邮箱/联系方式
- 订单来源
- 采购产品
- 产品成本
- 供应商
- 产品售价
- 美元售价
- 人民币售价
- 汇率
- 平台费用
- 物流费用
- 毛利润

台账只对管理员和主管账号显示，普通账号不显示。

这个模块的目标很明确：老板看利润，业务看订单，财务看收付款。

## 八、权限、审计和数据安全

随着系统越来越像真实 ERP，数据安全也必须考虑。

后来加入了登录界面和角色权限：

```text
admin   老板账号：看全部数据、管理用户、改配置、看审计日志
manager 主管账号：看全部业务数据、看台账，不能管理用户
普通账号：处理日常业务，不显示台账和审计日志
```

审计日志只给管理员看。

JSON 数据库也做了优化：

- 写入前备份
- 写入冲突检测
- 避免多人同时保存导致覆盖
- 保存成功/冲突都会写审计日志

虽然目前还是 JSON 数据库，但已经比最初安全很多。后续如果多电脑多人高频使用，最好迁移到 SQLite 或 PostgreSQL。

## 九、AI 和自动识别

这个 ERP 也尝试加入视觉模型识别：

- 上传采购单图片
- 上传销售单图片
- 上传报价表图片
- 自动识别内容并填入表格

视觉模型最初考虑过 OpenAI 和阿里云百炼，后来切换成阿里云百炼，主要考虑中文表格、价格表、国内网络环境和成本。

识别不是完全替代人工，而是先自动填，再人工核对。这个思路比较稳。

## 十、邮件功能

后面还加入了邮件模块。

支持：

- SMTP 配置
- 多收件人
- 附件
- 按收件邮箱归档
- 查看历史邮件内容

这个功能主要是为了把 PI、报价、单据直接从 ERP 发给客户，并保留发送记录。

## 十一、标准案例测试数据

做了一段时间后，我发现每次改功能，如果只靠真实数据测试，很容易漏问题，也不安全。

所以后来专门做了一套标准案例测试数据，覆盖：

- 供应商
- 商品
- 采购
- 销售
- 客户
- 报关单据
- 非报关单据
- 收付款
- 库存流水
- 台账
- 邮件历史

测试数据可以一键导入：

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\install-standard-case-data.ps1 -Apply
```

导入前会自动备份当前数据库。

这个标准案例非常有用，后续每次优化功能，都可以先跑一遍完整业务闭环。

## 十二、部署、GitHub 和 Obsidian

项目资料后来整理到了两个地方。

GitHub 用来保存部署包和代码：

```text
私有代码仓库
```

Obsidian 用来保存项目说明、部署记录、更新日志和使用说明：

```text
本地知识库目录
```

后面定了一个规则：

- GitHub 不上传真实业务数据
- 不上传 auth.json
- 不上传邮箱授权码
- 不上传真实单据
- 安装包更新必须带版本号
- Obsidian 记录重要更新，但不是每次小改都同步，除非明确要求

## 十三、目前最大的收获

这个项目让我最大的体会是：ERP 不是表单堆起来的，而是关系跑通的。

一个好用的小企业 ERP，重点是：

```text
少填
少错
能追溯
能汇总
能导出
出问题能查
```

比如销售单里一个产品，必须能知道它来自哪个供应商；客户单据必须能按订单和 CI No. 找到；收付款必须能对应客户、供应商或订单；库存变化必须有流水；利润必须能从台账里算出来。

这些关系一旦打通，系统就开始真正有用。

## 十四、后续可优化方向

后续还可以继续优化：

- JSON 数据库迁移到 SQLite 或 PostgreSQL
- 多用户并发进一步增强
- 单据 OCR 准确率继续提高
- 手机端页面适配
- 邮件模板管理
- 自动生成报关资料清单
- 自动检查订单缺哪些单据
- 自动生成利润分析报表
- 更细的角色权限
- 数据备份和恢复界面化

## 结语

这套定制ERP不是一开始就设计完整的，而是在真实业务里一点点长出来的。

从供应商登记、报价表、采购销售、库存、客户单据，到收付款、台账、审计、多电脑共享，每一步都是围绕实际问题改出来的。

现在它已经不只是一个“录数据的工具”，而是一个能帮助整理业务关系、减少重复输入、沉淀订单资料的小型业务系统。

对小公司来说，最适合自己的 ERP，往往不是最贵、最复杂的那个，而是最贴近自己业务流程、能持续优化的那个。
