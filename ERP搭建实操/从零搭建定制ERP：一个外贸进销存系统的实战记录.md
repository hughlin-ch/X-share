﻿# 从零搭建定制ERP：一个外贸进销存系统的实战记录

> 这是一篇关于“小型外贸业务如何从真实需求出发，逐步搭建一套定制 ERP”的项目复盘。它不是标准产品介绍，更像一次业务系统从 0 到 1 的实操记录。

## 背景

最开始的需求很朴素：我需要一套能登记供应商、管理采购和销售、统计收付款、管理库存、保存客户单据的系统。

市面上的 ERP 很多，但真正用起来，经常会遇到两个问题：

- 功能很全，但流程太重，很多字段并不适合自己的业务。
- 看起来能用，但客户单据、供应商报价、采购、销售、库存和收付款之间很难形成顺畅闭环。

所以我决定先做一套本地运行的轻量 ERP。目标不是一开始就做成“大而全”的系统，而是先贴合真实业务，把每天会用到的流程跑通，再在使用中逐步优化。

## 一、最初的目标

第一版目标很明确：先把核心业务录进去、查得到、算得出。

需要覆盖的基础能力包括：

- 供应商档案管理
- 商品与库存管理
- 采购入库、销售出库
- 应收、已收、应付、已付统计
- 客户档案和客户单据归档
- 多文件、多图片上传
- 采购单、销售单、报价表图片识别录入
- 本地运行，并支持通过内网工具给多台电脑访问
- 后期可以持续维护、迭代和优化

这个阶段最重要的不是界面漂亮，而是业务闭环完整。

## 二、从本地桌面系统开始

最早的版本是一个本地桌面 ERP：在 Windows 电脑上启动服务，通过浏览器打开本地访问地址。

后来考虑到办公室电脑、家里电脑可能都要用同一套数据，于是加入了内网访问方案。这样不需要把系统暴露到公网，也能让不同电脑访问同一台主机上的 ERP。

这个部署方式的优点是：

- 成本低，不需要购买复杂服务器。
- 数据在自己电脑上，掌控感更强。
- 多电脑访问可以通过内网工具实现。
- 后期如果业务量上来，也可以迁移到数据库和专门服务器。

## 三、模块从少到多，但核心始终是业务关系

系统最初只有几个基础模块，后来逐步扩展成完整结构。

目前主要模块包括：

- 经营总览
- 供应商管理
- 商品与库存
- 采购与销售
- 出入库流水
- 客户管理
- 客户单据
- 收付款管理
- 进项发票统计
- 台账
- 邮件
- 审计日志
- 系统自检

做了一段时间后，我越来越明确一件事：ERP 的关键不是页面数量，而是数据关系是否顺畅。

比如一张销售单，不能只记录“卖了什么”，还应该能继续追溯：

```text
销售单 → 客户 → PI → 产品 → 供应商 → 成本 → 收款 → 库存 → 台账利润
```

这些关系一旦打通，系统才会真正变成业务工具，而不只是电子表格的替代品。

## 四、供应商和报价表

供应商模块一开始只是登记基础信息，例如名称、联系人、电话和地址。

后面发现，采购真正需要的是“供应商和报价的长期沉淀”。于是系统加入了供应商报价表：每个供应商下面可以保存多份报价表，报价表可以上传原件图片，也可以把识别出的产品、型号、单价整理成表格。

后续又陆续优化了这些功能：

- 报价表支持自定义名称。
- 已有报价表可以修改名称。
- 报价原件支持上传多个文件。
- 报价产品搜索容错率更高。
- “货号”字段调整为更适合业务习惯的“型号”。
- 供应商界面直接显示应付、已付、未付金额。

这样供应商模块就不只是通讯录，而是采购价格和应付款的入口。

## 五、客户管理和客户单据

客户管理是后期改动最多的部分。

一开始，客户档案、客户合同、单据库是分散的。实际使用后发现，分散会让资料越来越乱，尤其是外贸订单涉及 PI、PL、CI、报关单、采购发票、提单、收汇水单等大量文件。

后来我把它们合并成一条关系：

```text
客户档案 → 客户合同 → 客户单据 → 订单 / CI No.
```

客户单据按业务类型分为：

- 报关订单
- 非报关订单

报关订单下面继续分组：

- 客户清关单据
- 国内清关单据
- 订单附件

客户清关单据通常包括：

```text
PI、PL、CI
```

国内清关单据通常包括：

```text
CI、PL、contract、国内采购合同、国内采购发票、国内运费发票、国际运费发票、报关单、报关委托书、放行书、出口退税联、报关费发票、提单、收汇水单、结汇水单、出口发票
```

最近还新增了“订单附件”分组，用来上传打包照片、发货照片、售后照片。

这个小功能很实用。很多售后问题最后都要回头查包装、装箱、发货现场照片，如果这些图片一开始就挂在订单下面，后续追溯会轻松很多。

## 六、采购、销售和库存闭环

采购和销售是 ERP 的主干。

采购入库需要记录：

- 供应商
- PI No.
- 产品名称
- 型号
- 颜色
- 数量
- 单价
- 总金额
- HS Code
- 税点
- 不含税金额
- 含税金额

销售出库需要记录：

- 客户名称
- 客户信息
- 币种，人民币或美元
- 产品名称
- 型号
- 颜色
- 数量
- 单价
- 总金额
- HS Code
- 产品对应供应商

销售单还可以自动生成 PI，并支持预览、修改、下载。

库存方面，系统会自动同步：

```text
采购入库 → 库存增加
销售出库 → 库存减少
手动调整 → 生成库存流水
```

后续为了减少重复录入，又做了业务录入闭环优化：

- 采购选择供应商后，商品下拉只显示该供应商的商品。
- 选择商品后，自动带出型号、颜色、HS Code、供应商、价格。
- 销售单每个产品行都保存供应商来源。
- 库存流水记录客户、供应商和业务日期。

这些细节看起来不大，但对后续查账、算利润、查供应商来源非常关键。

## 七、收付款和台账

收付款模块最初只是记录收入和支出，后来逐步细化成更接近财务视角的结构。

现在支持：

- 应收
- 已收
- 应付
- 已付
- 人民币和美元
- 客户或供应商
- 日期
- 多文件上传
- 关联订单
- 按客户、供应商、销售单分组
- 年/月汇总

台账模块则按 PI 汇总订单利润，主要字段包括：

- PI 号
- 客户名称
- 邮箱或联系方式
- 订单来源
- 采购产品
- 产品成本
- 供应商
- 产品售价
- 美元售价
- 人民币售价
- 汇率
- 平台费用
- 物流费用
- 毛利润

台账只对管理员和主管账号显示，普通账号不显示。

这个模块的定位很清楚：老板看利润，业务看订单，财务看收付款。

## 八、权限、审计和数据安全

随着系统越来越接近真实业务使用，权限和安全也必须补上。

后来系统加入了登录界面和角色层级：

```text
admin   老板账号：看全部数据、管理用户、改配置、看审计日志
manager 主管账号：看全部业务数据、看台账，不能管理用户
普通账号：处理日常业务，不显示台账和审计日志
```

审计日志只给管理员查看。

数据层也做了几项基础保护：

- 写入前自动备份。
- 写入冲突检测，避免多人同时保存时互相覆盖。
- 保存成功和保存冲突都会写审计日志。
- 重要资料支持导出和备份。

目前系统仍然是轻量本地数据库方案。对于小规模使用来说已经够用，但如果未来多人高频使用，下一步最好迁移到 SQLite 或 PostgreSQL。

## 九、AI 识别的加入

这个系统也尝试加入视觉识别能力，用来处理图片和文件录入。

主要场景包括：

- 上传采购单图片，自动识别采购内容。
- 上传销售单图片，自动识别销售内容。
- 上传报价表图片，自动识别产品、型号、单价。

这里的思路不是让 AI 完全替代人工，而是“先自动填，再人工核对”。

这样既能节省录入时间，又不会因为识别错误直接污染业务数据。

## 十、邮件功能

后面系统还加入了邮件模块。

主要支持：

- SMTP 配置
- 多收件人
- 附件发送
- 按收件邮箱归档
- 查看历史邮件内容

这个功能主要用于把 PI、报价、单据直接从系统发给客户，并保留发送记录。

## 十一、标准案例测试数据

做了一段时间后，我发现每次改功能，如果只靠真实业务数据测试，很容易漏问题，也不安全。

所以后来专门做了一套标准案例测试数据，用于回归测试。

测试数据覆盖：

- 供应商
- 商品
- 采购
- 销售
- 客户
- 报关单据
- 非报关单据
- 收付款
- 库存流水
- 台账
- 邮件历史

这套标准案例可以一键导入，导入前会自动备份当前数据库。

它的价值很大：后续每次改功能，都可以先跑一遍完整业务闭环，而不是拿真实业务数据冒险。

## 十二、Looping Engineering 的构想

做到后期，我开始把这个项目理解成一种“Looping Engineering”：不是一次性把系统设计完，而是围绕真实业务持续建立反馈循环。

这里的 loop 不是简单的反复修改，而是每次都让系统多形成一个闭环：

```text
业务输入 → 数据关联 → 自动汇总 → 异常检查 → 用户反馈 → 再优化
```

在这个项目里，主要形成了几类循环。

### 1. 业务录入闭环

最开始很多数据都要手工填，后来逐步变成“选一个关键对象，自动带出相关信息”。

比如：

- 选择供应商后，只显示该供应商的商品。
- 选择商品后，自动带出型号、颜色、HS Code、供应商和价格。
- 新建销售单后，自动进入 PI 生成和预览流程。
- 上传客户单据时，自动按合同、订单类型、CI No. 和资料分组归档。

这个循环的目标是减少重复输入，让业务人员把注意力放在核对和判断上，而不是机械复制。

### 2. 数据一致性闭环

ERP 最怕的是“看起来录进去了，但关系断了”。

所以后期专门加入了数据一致性检查：

- 商品是否绑定供应商。
- 销售单产品是否能追溯供应商来源。
- 库存数量是否和出入库流水一致。
- 收付款是否能对应客户、供应商或订单。
- 客户单据是否缺少合同、客户或 CI No. 关系。

发现问题后，不只是提示，还尽量提供修复入口。例如重建缺失库存流水、按流水校准库存、同步订单收付款等。

这类 loop 的核心是：系统不只负责保存数据，也要帮助发现数据哪里不可信。

### 3. 标准案例测试闭环

真实业务数据复杂、敏感，也不适合随便拿来测试。

所以后来建立了一套标准案例数据。每次新增功能或调整逻辑，都可以用这套数据跑一遍：

- 供应商和报价是否正常。
- 采购入库是否影响库存和应付。
- 销售出库是否影响库存和应收。
- 客户单据是否按订单分组。
- 台账是否能算出成本、费用和毛利润。
- 收付款筛选和汇总是否正确。

这样每次优化都不再只是“感觉能用”，而是有一条固定路线去验证。

### 4. 文档和部署闭环

项目越做越大后，只靠记忆维护是不现实的。

所以我把说明文档、部署记录、更新日志、标准案例、安装包规则都整理出来。每次重要更新后，再决定是否同步到代码仓库和本地知识库。

这个循环解决的是“下次还能不能接得上”的问题。

如果一个系统只能当前这次改得动，下次重新打开又要从头理解，那它就还没有进入可持续维护状态。

### 5. 人机协作闭环

这个项目还有一个很明显的特点：很多优化不是一开始就想清楚的，而是在使用、反馈、再实现的过程中冒出来的。

比如客户单据一开始只是上传文件，后来才发现需要按报关/非报关、客户清关/国内清关分类；再后来又发现打包照片应该单独作为订单附件保存，方便售后备查。

这类需求很难靠一次需求文档完全写出来。更有效的方式是：

```text
先做可用版本 → 真实使用 → 发现摩擦 → 小步优化 → 再回到真实使用
```

这就是我理解的 Looping Engineering：用持续的小循环，把一个工具慢慢打磨成贴合业务的系统。

## 十三、部署、代码和知识库管理

项目资料后来分成两类管理：

- 代码仓库：保存部署包、代码和脱敏测试数据。
- 本地知识库：保存项目说明、部署记录、更新日志和使用说明。

同时约定了几条安全规则：

- 不上传真实业务数据库。
- 不上传登录账号文件。
- 不上传邮箱授权码。
- 不上传真实业务单据。
- 安装包更新必须带版本号。
- 重要更新写入知识库，但小改动不一定同步。

这个过程让我意识到：项目能持续迭代，文档和备份非常重要。

## 十四、最大的收获

这个项目最大的体会是：ERP 不是表单堆起来的，而是关系跑通的。

一个好用的小企业 ERP，核心应该是：

```text
少填
少错
能追溯
能汇总
能导出
出问题能查
```

比如：

- 销售单里的每个产品，都应该知道来自哪个供应商。
- 客户单据必须能按订单和 CI No. 找到。
- 收付款必须能对应客户、供应商或订单。
- 库存变化必须留下流水。
- 利润必须能从台账里算出来。

这些关系一旦打通，系统就开始真正有用了。

## 十五、后续优化方向

后面还可以继续优化：

- 把轻量数据存储迁移到 SQLite 或 PostgreSQL。
- 增强多用户并发能力。
- 提高单据 OCR 准确率。
- 优化手机端页面适配。
- 增加邮件模板管理。
- 自动生成报关资料清单。
- 自动检查订单缺哪些单据。
- 自动生成利润分析报表。
- 设计更细的角色权限。
- 把数据备份和恢复做成界面化操作。

## 结语

这套定制 ERP 不是一开始就设计完整的，而是在真实业务里一点点长出来的。

从供应商登记、报价表、采购销售、库存、客户单据，到收付款、台账、审计、多电脑共享，每一步都是围绕实际问题改出来的。

现在它已经不只是一个“录数据的工具”，而是一个能帮助整理业务关系、减少重复输入、沉淀订单资料的小型业务系统。

对小公司来说，最适合自己的 ERP，往往不是最贵、最复杂的那个，而是最贴近自己业务流程、能持续优化的那个。
